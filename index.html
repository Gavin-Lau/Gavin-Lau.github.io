go-11 包管理工具 go mod

Posted on 2020-05-26 | In go

go mod开启

至于网上说的这个命令

我理解是对于使用go get 命令下载不到的会尝试使用此代理吧，比如golang的x库

还有其他几个命令的具体含义

go mod 常用命令

只列出几个常用的命令

  1. go mod init 初始化modules，使用这个命令以后，会创建go.mod 文件
  2. go mod download 下载依赖包到 $GOPATH/pkg 目录下
  3. go mod tidy 在修改了依赖包，或者版本以后，使用该命令会重新检查依赖关系，删除无用的，添加缺失新建的包
  4. go mod vendor 生成vendor目录，将依赖复制到vendor目录下面
  5. go list -m all 查看当前包的所有依赖关系
  6. go list -m -versions {package-name} 查看名字叫 package-name的包的所有可用版本

go mod 的优点

  1. 以前go的所有项目只能在$GOPATH/src/ 下去创建相应的包，目录是死的,使用了go module之后，可以彻底摆脱$GOPATH
  2. 以前govendor，godep，各种版本的包管理工具林立，且不怎么好用，go mod 增加了对版本的支持
  3. 使用go mod以后，go run、go build等工具链的包依赖关系都会交给go mod 去管理

总之，go mod的出现，是对go的包依赖关系的解决方案的唯一正解

context的使用场景

与goruntime配合使用，也就是说他用在单业务跨多协程，保持各个协程高度一致的作用，当上层的goruntime执行失败时，可以通过context传递信息到底层的goruntime，尽量早的释放资源。

Context的底层结构

主要对于这4个接口的理解，对于context的理解和使用

  • Deadline: 返回当前context的任务将要被取消的deadline，如果没有deadline，ok 返回false
  • Done: 当前context被取消时，将返回一个关闭的channel，这里的利用的channel的特性:

    如果channel c已经被关闭,继续往它发送数据会导致panic(send on closed channel),但是从这个关闭的channel中不但可以读取出已发送的数据，还可以不断的读取零值

  • Err 配合Done使用，在Done返回后，Err告诉程序猿Done的原因
  • Value 用于同一个上下问中，在不同的goruntime中传递特定的数据，是一个键值队，所以要通过一个key获取对应的value，线程安全

signal

1. unix信号

典型的SIGINT、SIGTERM，os.signal包的用途就是改变这些信号的默认动作

2. 实现方式

golang推荐使用chan来实现

3. 几个重要的包内

  • Notify:改变信号的默认行为
  • Ignore: 忽略某些信号,什么都不处理
  • Reset: 重置信号的行为到默认
  • Stop: 停止接收信号，但是并没有重置默认行为。TODO:need more info

4. 例外

程序无法捕获信号 SIGKILL 和 SIGSTOP （终止和暂停进程），因此 os/signal 包对这两个信号无效。
